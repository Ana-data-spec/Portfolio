Time-to-Attribution: Estimating Days from Coverage Start to First PCP Assignment

An active analysis of how long it takes a new health plan member to be attributed to a Primary Care Provider after coverage begins — and which member segments are likely to attribute fastest or slowest.

Active
PROJECT STATUS
6–24mo
ROLLING ATTRIBUTION WINDOWS
Python
PANDAS, NUMPY, MATPLOTLIB

The Problem

Health plans use claims-based logic to attribute members to a Primary Care Provider (PCP), typically based on visit patterns within a rolling window (6–24 months). The speed of that attribution matters: members who go unattributed for long stretches can fall outside care coordination programs and may be miscounted in quality measures like HEDIS. This project asks a simple operational question: how long does attribution actually take, and for whom does it take longest?

Approach

This analysis is currently in progress. The planned approach:

  • Build a dataset of members with coverage start dates and claims-based attribution events, capturing the rolling-window attribution logic (6/12/18/24 months) used in production
  • Calculate days-to-attribution per member and analyze the full distribution, not just an average, since attribution timing is expected to be right-skewed
  • Segment results by factors likely to affect attribution speed — plan type, region, and age group — to identify where attribution lags
  • Surface cumulative attribution rates at key milestones (30/60/90/180 days) to make the findings actionable for care coordination teams

Planned Output

The final deliverable will include a distribution analysis, a cumulative attribution curve, and segment-level comparisons (by plan type and region), built in Python. This page will be updated with findings once the analysis is complete.

Status: This project is in progress. The method and framing above reflect real, ongoing member attribution work; findings will be added here once finalized. No real member, claims, or employer data is used or referenced in any published version of this analysis.

Parameterized Data Quality Profiling Tool (Databricks + HTML)

A reusable two-part profiling tool: a parameterized Databricks notebook that generates a native dashboard for any input table, plus a dedicated HTML view built specifically for attribution data profiling.

The Problem

Large claims and encounters datasets accumulate quality issues silently — a missing diagnosis code, an inconsistent eligibility span, a malformed provider ID — that don't surface until a downstream report breaks or a HEDIS measure comes back wrong. Building a one-off profiling check for a single table solves the problem once; building a reusable, parameterized tool solves it for every table that comes after.

Approach

Designed a two-part system rather than a single fixed dashboard:

  • Generic profiling notebook (Databricks): a parameterized notebook where a user enters any table name as input. The notebook runs profiling logic against that table and refreshes a native Databricks SQL dashboard — built using Databricks' own dashboard widgets — showing null rates, anomalies, and completeness for whatever table was specified, without needing to hardcode logic per dataset
  • Dedicated HTML view (attribution data): a custom-built HTML page specifically designed to display profiling results for member attribution data, since this dataset warranted a more tailored, stakeholder-facing presentation than the generic dashboard provides

This split reflects a common real-world tradeoff: a generic tool covers the long tail of tables that need occasional checks, while a hand-built view is worth the extra effort for the highest-priority dataset.

Status

The generic Databricks notebook and dashboard are functional and in active use. The HTML attribution view is built and serving as the dedicated front-end for that dataset specifically.

A K-means clustering model to detect fraudulent return behavior from transaction-level retail data. Awarded 1st place for analytical innovation.

The Problem

Retail return fraud — excessive returns, "wardrobing," and return-for-refund schemes — costs retailers billions annually, but most fraud isn't a single suspicious transaction; it's a pattern across many transactions from the same customer. The challenge: separate genuinely high-return customers from those exhibiting a distinct, suspicious behavioral signature.

Approach

Used unsupervised learning rather than fraud labels (which weren't available) to let the data reveal natural groupings of customer behavior.

  • Engineered features per customer: return frequency over a 90-day window and average return value
  • Applied K-means clustering to segment customers into behavioral groups
  • Identified a distinct cluster of high-frequency, high-value returners as the highest-risk segment for manual review

Findings

A clear behavioral cluster emerged with both high return frequency and high average return value — a combination far rarer among normal shoppers.
Of 860 customers analyzed, 60 (~7%) fell into the flagged segment — a manageable shortlist for manual fraud review rather than a blanket policy change.

Takeaway

Clustering surfaced a actionable, narrow list for investigation instead of a noisy rule-based flag system. This approach won 1st place in the competition for combining a simple, interpretable model with a result a non-technical fraud team could immediately act on.

U.S. University Diversity Visualization

The Problem

Enrollment diversity data is often published as dense tables that university administrators and policy stakeholders don't have time to parse. The goal was to translate multi-year, multi-institution enrollment data into visuals a non-technical audience could read in seconds, not minutes.

Approach

Pulled and cleaned multi-year enrollment data by race/ethnicity across U.S. institutions, then built two complementary views: a year-over-year shift comparison and a regional diversity index using a Simpson Diversity Index calculation (a standard measure of how evenly distributed a population is across categories).

  • Built interactive dashboards in both Tableau and Power BI to compare tool tradeoffs for this audience
  • Calculated a Simpson Diversity Index per region to quantify "how diverse," not just "what % of each group"
  • Designed visuals specifically for non-technical readers — direct labels, minimal jargon, clear before/after framing

Findings

Hispanic and Asian student enrollment share grew the most between 2015 and 2023, while the White student share declined by roughly 9 percentage points.
The West region showed the highest diversity index, while the Midwest showed the lowest — useful context for institutions benchmarking against regional peers rather than national averages alone.

Takeaway

A single index number plus a clear before/after chart communicated more to stakeholders in one meeting than the full underlying dataset would have in a week of follow-up emails — a reminder that the right visualization is itself a form of analysis.
